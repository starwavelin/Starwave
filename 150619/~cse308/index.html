Software Engineering Project: Requirements of Undergraduate Majors

Requirement Analysis

Goal: What is Requirements of Undergraduate Majors (RUM) system?

It is a system that

Helps undergraduate students determine whether they are on track to satisfy the major requirements in a timely fashion.

  • For example, a student can submit a major and course plan, which includes courses they have taken and courses they plan to take, and the system will check all the requirements for that major and indicate which requirements, if any, are not satisfied by the course plan.

Helps departmental administrative staff determine whether to clear a student for graduation.

  • For example, when a student is ready to graduate, the department's undergraduate secretary can export the student's course information from the registrar's information system and then import it into this system as a course plan, and the system will indicate which quirements, if any, are not satisfied.

Detailed Requirements for Different Majors

The 4 Stony Brook University majors that our project will support:

  • CSE--computer science
  • ISE--Information Systems
  • MEC--Mechanical Engineering
  • ESE--Electrical Engineering

Specializations within a major

  • For the convenience, specializations of a major would be counted as different majors. For instance, information security specialization of the computer science major would be viewed different from the regular CSE track. We may assign a major code "CSEIS" for this specialization

Requirement Version

  • example: 2010 version.

Requirements: 3 kinds

  1. Course requirements ----courses must be taken
  2. GPA requirements-----a minimum GPA for a set of courses
  3. Residency requirements------require that some courses be taken at Stony Brook University (as opposed to transfer of credits for an equivalent course at another institution)

Course Offerings

System maintains information about courses offered in each semester.

  • Course
    • Examples: CSE 308, MAT 125
  • Semester
    • (Fall, Winter, Spring, SummerI, or SummerII) and the year.
  • Status (provide tentative information about likely course offerings)
    • (definite or tentative)

Student Information

  • Major
  • Requirement year (Matriculation year)
    • The year the student entered the university
    • The year of the student's most recent return from a leave of absence
  • Requirement version (The version of the degree requirements that the student wishes to satisfy
    • requirement version >= requirement year
  • Course Plan
    • Set of entries representing courses the student has taken, is currently taking, or plans to take.
    • Each entry contains:
      • Course
        • example: CSE308
      • Semester
        • a semester (e.g. Fall 2010) or a special value (e.g. null) meaning "unspecified"
      • Suggested (true or false)
        • True -- System suggested some information in this entry, and the student has not yet confirmed its acceptability
        • False -- Student provided all of the information; the system suggested the information and student confirmed it.
  • Grade
    • A grade
    • A value (e.g. null) indicating the grade is unknown yet
  • Transfer (true or false)
    • True -- An approved equivalent course was taken at another institution
    • False -- The course was taken at Stony Brook

Maintaining Student Information and Checking Requirements

  • Display student information
    • Student information (current session)
    • Course plan
      • Tentative course offerings should be visibly distinguished from definite course offerings
      • The course offering information does not indicate that the course will be offered that semester should be visibly distinguished from other entries. (System's course offering information may be incomplete.)
    • Edit student information
      • Current session
      • The system stores this information until the user terminates the session
      • If student information has not been uploaded in this session, editing starts from "empty" student information
    • Download student information
      • Current session
      • Download student information, and uploading it next time he/she uses the system
    • Upload student information
      • Reads a file containing student information
        1. Import csv-format transcript. (Courses a student has taken or is taking, course plan associated with the current session, grade (may be empty))
          • examples: Fall 2009,BIO 201,A-
        2. Import SOLAR-format transcript (Extra credit) (HTML file produced by using SOLAR)
    • Check requirement
      • Satisfied or not satisfied.
      • Users request check requirement, or system continuously updates the display to reflect this information.
      • The system give feedback about unsatisfied requirements. These unsatisfied requirements may include:
        1. E.g. relevant courses were not taken
        2. E.g. grade was too low.
        3. E.g Course plan contains sufficient courses to satisfy a particular requirement, but the grades for some of those courses are unknown, the system should indicate the status of the requirement as "pending" and indicate the minimum grades that will be needed in those courses for the requirement to be satisfied.
          • For Course requirements ,"CS 2009" and "CS 2009: 2. Required Advanced Courses: part 1" are "expected to be satisfied" under the condition that the grade in CSE 308 is a C or higher.
          • For a GPA requirement, the status of the GPA requirement should be reported as "pending" (not "unsatisfied").
    • Suggest schedule
      • Assign semesters to entries with semester==null in the course plan, consistent with the course offerings
      • Sets "suggested" to true for those entries.
      • Never schedules in course plan that are not listed as definitely or tentatively scheduled, although the user may do that.
      • The suggested schedule should be consistent with course prerequisites.
      • Minimize
        1. The number of semesters until all requirements are satisfied
        2. The number of entries with tentatively scheduled courses.
    • Exit
      • Student information associated with the current session is discarded.

Administration of Course Offerings and Course Prerequisites

  • Load course offerings
    • Reads from a specified file, or from files in a specified directory.
    • Adds the information to its database.
    • The system does this at startup, and an administrator may invoke this function later.
      • examples: Fall 2010,CSE 308
  • Delete all course offerings
    • Deletes all information from the in-memory course offering database.
    • Administrator calls this function before loading updated course offering information.
  • Delete specified course offerings
    • Deletes course offering information for a specified semester or a specified subject.
    • Administrator calls this function before loading updated course offering information for a semester.
  • Load course prerequisites
    • Reads from a specified file, or from files in a specified directory
    • Adds the information to its database
    • The system does this at startup, and an administrator may invoke this function later, while the system is running
      • Example: CSE 312, (U3|U4)
      • Example: CSE 305, ((CSE 219|CSE 260)&CSE 220)

ADMINISTRATION OF REQUIREMENTS FOR MAJORS

  • FILE-BASED ADMINISTRATION or
    • load requirements
      • Loads information from a specified file, or from files in a specified directory, into the database.
      • Delete existing requirements for major. Insert new requirements.
      • The file should not contain code in a conventional programming language.
    • Delete all requirements
    • Delete requirements for specified major
  • GUI-BASED ADMINISTRATION
    • Creating, editing, and deleting requirements for majors
    • Allow the user to create a new version of requirements from an existing version
    • The system stores the information in a persistent database or a file (which does not need to be human-readable)
    • When the system starts up, it reads all requirements into its in-memory database

OTHER SYSTEM REQUIREMENTS

  • System security
    • Administrative functions (i.e., functions related to course offerings and requirements for majors) should be password-protected.
  • Network security
    • Use of secure communication is not required.
  • Concurrency
    • Multiple users may use the system concurrently, to enter student information, check requirements, etc.
    • Administrative functions may be invoked while other users are using the system.
    • Synchronization should be used to ensure reasonable behavior.
  • No arbitrary limits
    • For example:
      • No limit on the number of entries in a course plan
      • No limit on the number of courses involved in a course requirement.
